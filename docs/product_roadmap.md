# PK ModelApp: продуктовый roadmap

Обновлено: 2026-07-09

Этот файл хранит продуктовые задачи вне чата, чтобы после каждого завершенного шага мы не искали следующий пункт в истории переписки.

## Как ведем roadmap

- После каждого завершенного шага обновляем соответствующий раздел: переносим пункт из “Не хватает” в “Уже есть/Частично есть” или уточняем следующий шаг.
- Если шаг был закоммичен, указываем его в журнале изменений ниже.
- Roadmap не заменяет технические TODO в коде, но хранит продуктовую последовательность работ.

## 1. Медиа

Уже есть:
- Загрузка фото и видео в анкеты.
- Превью медиа, выбор главного фото, модерация анкеты.
- Постоянная очередь загрузок со статусами и повтором на карточке анкеты.
- Подготовка видео перед upload на iOS и desktop, где это поддерживается.
- Сжатие и ресайз фото перед upload.
- Реальный progress upload по файлам в очереди.
- Пауза текущего upload с отменой активного запроса и retry незавершенного файла.
- Восстановление Web-очереди после перезапуска браузера через IndexedDB для выбранных файлов.
- OPFS/File System Access хранение больших Web-файлов с fallback на IndexedDB.
- UI-диагностика очереди: источник хранения, восстановление, ошибки и retry-состояния по файлам.
- Thumbnail pipeline для видео: Web/iOS/desktop генерируют и сохраняют JPEG-обложку перед публикацией видео.
- Нативное восстановление исходного Web-файла через File System Access handle там, где браузер дает постоянное разрешение, с fallback на OPFS/IndexedDB.
- Настоящий resumable upload для больших видео через Supabase Storage TUS: upload URL и offset сохраняются в очереди, после паузы/перезапуска загрузка продолжается чанками.
- Стабилизирована Web-загрузка медиа на GitHub Pages/desktop Chrome: File System picker читает JS-массив handles через индекс, без ошибки `m.item is not a function`.
- Улучшена диагностика сохранения медиа: технические ошибки показываются в редакторе, а runtime-ошибки после запуска больше не заменяют приложение экраном “Ошибка запуска”.
- Нормализована связка `video_urls` + `video_preview_urls`: массивы приводятся к одинаковой длине, чтобы превью не съезжали относительно видео.
- В редакторе анкеты видео открывается и запускается из самой плитки, а превью не блокирует воспроизведение.

Не хватает:
- Chunk/resumable upload для остальных типов больших файлов, если появится такая продуктовая необходимость.

## 2. Профиль модели

Уже есть:
- Анкета, обложка, галерея фото/видео, просмотр медиа, модерация, дата рождения/возраст, типы анкет.
- Категории медиа для фото/видео.
- Отдельный showreel-блок и отметка showreel в видео.
- Видео в редакторе анкеты открываются из медиагалереи и запускаются как полноценный preview/player.
- Выбор главного фото через `cover_photo_url`.
- Автоматический face-aware focal point для обложки: при выборе главного фото приложение пытается найти лицо и сохранить `cover_photo_focal_x/y`.
- Категория `Full length` заменена на `Снэп-шоты`.
- Для актеров включены физические параметры как у моделей: дата рождения/возраст, рост, грудь/талия/бедра, цвет глаз/волос.
- Multi-role анкета: отдельное поле `profile_roles text[]` и UI “добавить роль”, чтобы одна анкета могла быть Модель + Актер + Фотограф, а блоки полей добавлялись по ролям и не заменяли друг друга.
- Каталог фильтруется вкладками по ролям: multi-role анкета отображается в каждой вкладке своей роли.
- Компактный selector ролей в каталоге, быстрый сброс активных фильтров без входа в расширенный поиск и сохранение текущего поискового набора как пресета.
- Сохраненные поиски в каталоге были реализованы с Supabase-backed storage, но временно выключены в UI из-за повторного зависания загрузки пресетов; код и таблица оставлены для отдельного возврата.

- Журнал действий можно чистить выборочно: галочки по записям, выбор всех видимых строк и удаление выбранных событий без общей опасной очистки всего журнала.
- Отдельная Web-first таблица пользователей в админке: поиск по имени/email/телефону/tag, фильтр ролей, контакты, город, активность и быстрый переход к чатам.
- Отдельная Web-first таблица всех анкет в админке: поиск по имени/городу/владельцу, фильтры статуса и ролей, владелец, параметры, медиа-счетчик и быстрый переход в анкету или модерацию.
- Отдельные Web-first таблицы всех кастингов и всех подборок в админке: поиск, фильтры этапов/статусов/публичности, счетчики откликов/анкет/референсов и быстрый переход в рабочие страницы.
- Back-office действия в отдельных таблицах: админ может удалить анкету, кастинг, подборку, профиль аккаунта, открыть чат пользователя, выдать/снять админку и переключить аккаунт между обычным пользователем и заказчиком.

Не хватает:
- Продвинутых массовых workflow: сохранение детального журнала попыток bulk-операций как отдельной сущности, повтор неудачных операций из истории, эскалации и уведомления по нарушению SLA.
- Расширения audit на будущие back-office зоны: пользователи, жалобы/споры, billing, ручные правки тарифов и роли.
- Отложено: детальная карточка аккаунта внутри back-office с историей, связанными анкетами/кастингами и действиями оператора; это не берем в первый запуск, чтобы не перегружать админку.

## 11. Безопасность и приватность

Уже есть:
- Supabase Auth, RLS-политики, роли аккаунтов, разграничение admin/casting/user.
- Email/phone sign-in и подтверждение email/телефона в аккаунте.
- Удаление аккаунта и базовые safety/admin-зоны.
- Auth-ошибки входа локализуются: `Invalid login credentials` показывается пользователю по-русски как неверный email/телефон или пароль.
- Публичные launch-draft документы: Privacy Policy, Terms of Service, Cookie Policy и Consent/Data Processing Notice доступны как отдельные страницы `/privacy`, `/terms`, `/cookies`, `/processing-notice`.
- Принятие документов при email/phone регистрации: пользователь явно принимает consent, версии документов и дата принятия сохраняются в auth metadata; SQL/RPC `legal_consents.sql` подготовлен для отдельной таблицы `user_legal_consents` после применения в Supabase.
- Проверка силы нового пароля при регистрации/смене: минимум 8 символов, буквы + цифры/символы, запрет популярных, повторяющихся и последовательных паролей, запрет включать email/телефон; в UI есть шкала надежности и подсказки. Логин сохраняет старую мягкую проверку, чтобы не заблокировать существующие короткие пароли.
- Мягкий anti-bruteforce/cooldown в клиенте: email/phone login блокируется после серии неудач, phone OTP и resend email verification имеют cooldown и временную блокировку после частых попыток.
- Security notifications: новый вход, смена пароля, запрос смены email и смена телефона создают системное уведомление через `app_notifications`, попадают в центр уведомлений и используют существующую push/email delivery-очередь по настройкам пользователя.
- Первый экран управления устройствами: “Устройства и входы” показывает текущую сессию и push-устройства из `push_device_tokens`, дает отключить push для устройства и выйти на текущем устройстве.
- Data export и удаление персональных данных: отдельная страница “Данные и удаление” собирает JSON-экспорт доступных данных аккаунта в буфер обмена и объясняет процедуру удаления через `delete_my_account`.
- Чувствительные чатовые медиа частично защищены: новые фото/видео/файлы/голосовые сообщения сохраняются в private bucket `chat-media`, в БД хранится `storage://` ссылка, показ идет через short-lived signed URLs; старые public URLs остаются совместимыми.

Не хватает:
- Юридической финальной вычитки публичных документов перед масштабным запуском.
- Применить `supabase/sql/legal_consents.sql` в Supabase, чтобы consent дополнительно писался в отдельную таблицу; IP/User-Agent на первом этапе фиксируются как client hint/пустой IP, полноценный серверный capture можно усилить отдельным шагом.
- Настоящий server-side rate limiting для Auth через Edge Function/прокси-слой или отдельную серверную схему, если перед production потребуется защита сильнее локального UI cooldown.
- 2FA/MFA для админов и casting/account-owner ролей, с резервными кодами.
- Полного server-side управления Auth-сессиями: список всех Supabase Auth sessions/devices, “выйти со всех устройств”, принудительный revoke старых refresh tokens.
- Расширить security notifications на подозрительную активность и будущий список активных сессий, когда появится server-side источник этих событий.
- Приватности профиля и account tag: публичный/только участникам диалогов/скрытый режим, отдельные настройки видимости email/телефона.
- Audit trail для чувствительных действий пользователя: смена контактов, пароля, ролей, удаление аккаунта, экспорт данных.
- Улучшить экспорт до настоящего download-файла/архива с медиа, если это понадобится после первого запуска; текущий launch-first вариант копирует JSON.
- Завершить защиту медиа вне чатов: аккуратно перевести приватные/непубличные фото анкет, подборок и avatar/account media на signed URLs без поломки публичного каталога, PDF и клиентских подборок; добавить watermark/anti-scraping там, где уместно.
- Content moderation/safety flow для жалоб, блокировок пользователей и скрытия подозрительных аккаунтов.
- Admin security hardening: MFA обязательно, ограничение опасных действий, reason/comment для ручных правок, журнал доступа.
