# PK ModelApp: продуктовый roadmap

Обновлено: 2026-07-08

Этот файл хранит продуктовые задачи вне чата, чтобы после каждого завершенного шага мы не искали следующий пункт в истории переписки.

## Как ведем roadmap

- После каждого завершенного шага обновляем соответствующий раздел: переносим пункт из “Не хватает” в “Уже есть/Частично есть” или уточняем следующий шаг.
- Если шаг был закоммичен, указываем его в журнале изменений ниже.
- Roadmap не заменяет технические TODO в коде, но хранит продуктовую последовательность работ.

## 1. Медиа

Уже есть:
- Загрузка фото и видео в анкеты.
- Превью медиа, выбор главного фото, модерация анкеты.
- Постоянная очередь загрузок со статусами и повтором на карточке анкеты.
- Подготовка видео перед upload на iOS и desktop, где это поддерживается.
- Сжатие и ресайз фото перед upload.
- Реальный progress upload по файлам в очереди.
- Пауза текущего upload с отменой активного запроса и retry незавершенного файла.
- Восстановление Web-очереди после перезапуска браузера через IndexedDB для выбранных файлов.
- OPFS/File System Access хранение больших Web-файлов с fallback на IndexedDB.
- UI-диагностика очереди: источник хранения, восстановление, ошибки и retry-состояния по файлам.
- Thumbnail pipeline для видео: Web/iOS/desktop генерируют и сохраняют JPEG-обложку перед публикацией видео.
- Нативное восстановление исходного Web-файла через File System Access handle там, где браузер дает постоянное разрешение, с fallback на OPFS/IndexedDB.
- Настоящий resumable upload для больших видео через Supabase Storage TUS: upload URL и offset сохраняются в очереди, после паузы/перезапуска загрузка продолжается чанками.
- Стабилизирована Web-загрузка медиа на GitHub Pages/desktop Chrome: File System picker читает JS-массив handles через индекс, без ошибки `m.item is not a function`.
- Улучшена диагностика сохранения медиа: технические ошибки показываются в редакторе, а runtime-ошибки после запуска больше не заменяют приложение экраном “Ошибка запуска”.
- Нормализована связка `video_urls` + `video_preview_urls`: массивы приводятся к одинаковой длине, чтобы превью не съезжали относительно видео.
- В редакторе анкеты видео открывается и запускается из самой плитки, а превью не блокирует воспроизведение.

Не хватает:
- Chunk/resumable upload для остальных типов больших файлов, если появится такая продуктовая необходимость.

## 2. Профиль модели

Уже есть:
- Анкета, обложка, галерея фото/видео, просмотр медиа, модерация, дата рождения/возраст, типы анкет.
- Категории медиа для фото/видео.
- Отдельный showreel-блок и отметка showreel в видео.
- Видео в редакторе анкеты открываются из медиагалереи и запускаются как полноценный preview/player.
- Выбор главного фото через `cover_photo_url`.
- Автоматический face-aware focal point для обложки: при выборе главного фото приложение пытается найти лицо и сохранить `cover_photo_focal_x/y`.
- Категория `Full length` заменена на `Снэп-шоты`.
- Для актеров включены физические параметры как у моделей: дата рождения/возраст, рост, грудь/талия/бедра, цвет глаз/волос.
- Multi-role анкета: отдельное поле `profile_roles text[]` и UI “добавить роль”, чтобы одна анкета могла быть Модель + Актер + Фотограф, а блоки полей добавлялись по ролям и не заменяли друг друга.
- Каталог фильтруется вкладками по ролям: multi-role анкета отображается в каждой вкладке своей роли.
- Компактный selector ролей в каталоге, быстрый сброс активных фильтров без входа в расширенный поиск и сохранение текущего поискового набора как пресета.
- Сохраненные поиски в каталоге: кнопка сохранения появляется только для нового активного набора фильтров, сохраненные пресеты можно быстро применить, переименовать или удалить.
- Композитки/PDF прямо из профиля: генерация PDF-портфолио с главным фото, сеткой фото, данными анкеты, резюме и публичной ссылкой.
- Подача профиля как портфолио: верхний portfolio-блок с крупным медиа, именем, ролями, ключевыми параметрами и быстрыми действиями для PDF/ссылки/showreel.
- Расширенные CTA в одном месте: пригласить, добавить в подборку/папку/избранное, написать в чат с контекстом анкеты.
- Полировка CTA “пригласить”: выбор конкретного кастинга, редактируемый шаблон сообщения, запись статуса `invited` в откликах кастинга и компактная история приглашений в профиле.
- Общая история быстрых действий в профиле: приглашения, добавления в подборки/папки и последние сообщения по анкете в одной компактной ленте.
- Audit log быстрых действий: отдельная таблица `profile_action_logs`, snapshot автора, шаблоны приглашений, связанные сущности и статусы `created/sent/delivered/read/failed`.
- Расширенный UI истории действий: фильтры по типам, просмотр полного текста шаблона/сообщения, timeline доставки и отдельная страница audit log для админки.
- Поиск/экспорт audit log и глубокая фильтрация по источнику, типу, автору, анкете/связи и дате.
- Серверное обновление статусов доставки/прочтения для email/push вне чата: `app_notifications` может быть связан с `profile_action_logs`, а SQL-триггер переносит `sent/delivered/read/failed` в историю действий профиля.
- Production delivery worker `send-notifications`: единая Supabase Edge Function для push через FCM и email через Resend, с записью серверных статусов обратно в `app_notifications`.
- Firebase/Resend secrets в Supabase частично заведены; домен отправителя для production email оставлен открытым до покупки и подключения домена.

Не хватает:
- Включенного production расписания/webhook для `send-notifications`.
- Production `EMAIL_FROM` на подтвержденном домене Resend. Сейчас этот пункт осознанно отложен до покупки домена.

## 3. Chats

Уже есть:
- Диалоги между аккаунтами.
- Контекст анкеты/кастинга.
- Media/file attachments.
- Read/typing, pinned/archive состояния.
- Desktop split-view.
- Reply на сообщение через цитату в composer и bubble.
- Выделение собственных сообщений, мультивыбор и удаление выбранных сообщений с подтверждением.
- Поиск по сообщениям внутри открытого чата: панель поиска, счетчик совпадений, переход по найденным сообщениям и подсветка найденного текста.
- Закрепленные сообщения внутри чата: pin/unpin из действий сообщения, компактный блок закрепов над перепиской и переход к сообщению, если оно уже загружено.
- Отдельный quick emoji/reaction UI убран из активного чата, чтобы не дублировать системную клавиатуру и не перегружать MVP.
- Голосовые сообщения: запись, предпрослушивание, отправка, player в bubble.
- Голосовые визуально доведены ближе к мессенджеру: waveform, длительность, статус внутри bubble.
- В списке чатов последнее голосовое показывается как voice-preview: waveform, длительность, “новое” при unread.
- Настоящее состояние прослушивания голосовых: `listened_at`, RPC `mark_selection_chat_audio_listened`, статус “прослушано” отдельно от “прочитано”.
- Voice UX проверен на уровне сборок: `flutter build web --release --base-href /PKModelApp/ --no-pub` и `flutter build ios --simulator --no-pub`.
- Production Web для voice QA открывается до `/login` на `https://app.pk.management/`; полный сценарий записи/playback/listened требует входа в тестовый аккаунт с чатом.
- Серверный поиск по всей истории чата: RPC `search_selection_chat_messages`, debounce в UI, поиск без предварительной подгрузки старых сообщений и переход к найденному сообщению.
- Глубокие фильтры списка чатов: новые, закрепленные, медиа, файлы, голосовые; строка поиска по чатам дополнительно ищет совпадения в сообщениях всех диалогов.
- Пересылка сообщений между диалогами: одиночная пересылка из меню сообщения, массовая пересылка из режима выбора, сохранение текста/медиа/файлов/голосовых и метка “Переслано”.
- Заголовок диалога строится по ФИО аккаунта собеседника, а не по компании; account tag участвует в поиске по списку чатов.

Не хватает:
- Ручного живого QA voice UX на реальном iPhone: подключить устройство по USB/исправить CoreDevice destination, затем подтвердить микрофон, playback и `прослушано` на production data.
- Полного Web voice QA после входа в тестовый аккаунт с доступным чатом: запись, отправка, playback и отметка `прослушано`.

## 8. Доверие и статус

Уже есть:
- Статусы аккаунта, заявки заказчиков, модерация анкет, verification SQL/поля.
- Отклоненная заявка на статус аккаунта показывается один раз на конкретную заявку через `rejection_seen_at`.
- Account tag в профиле аккаунта: пользователь задает публичный handle вроде `@artemkukhar`, каталог показывает tag вместо email, если он заполнен.

Не хватает:
- Видимых бейджей.
- Истории изменений статуса.
- Timeline-модерации.
- Понятных причин правок в красивом UI.
- Публичной карточки аккаунта по tag: `/@artemkukhar` с аватаром, компанией, ролями, городом, ссылками и кнопкой “написать”.
- Умных mentions в чате: ввод `@art...`, выбор аккаунта, mention-уведомление адресату.
- Верифицированных бейджей аккаунтов: verified agency / casting director / model parent.
- Расширенной истории статуса аккаунта: когда подал заявку, кто отклонил/одобрил, причина и что исправить.
- Настроек приватности handle: показывать tag всем, только участникам диалогов или скрывать.

## 10. Админка

Уже есть:
- Модерация, заявки агентов, подборки, кастинги, safety, merge requests.
- Audit log быстрых действий по анкетам с фильтрами, деталями и timeline.
- Back-office audit log: отдельная таблица `admin_action_logs` и запись ключевых действий админа — модерация анкет, заявки заказчиков, создание/удаление кастингов.
- Расширенный back-office audit для пользовательских заявок и операций: заявки на объединение аккаунтов, safety-жалобы, статусы/публичность подборок и массовое удаление подборок/кастингов.
- Единая стартовая back-office панель: сводка входящих задач, группировка “очередь/операции/контроль”, бейджи по категориям и общий бейдж на вкладке “Админ”.
- Операторский стол внутри админки: единая таблица очередей по анкетам, заявкам и safety с поиском, фильтрами и выбором строк.
- Bulk-действия в операторском столе: массовое одобрение/отклонение анкет, закрытие safety-жалоб, CSV-экспорт выбранных строк и запись результата в `admin_action_logs`.
- Подтверждения перед опасными bulk-действиями и построчный результат выполнения: успешно/ошибка, причина ошибки и повтор только неудачных строк.
- Назначение ответственного оператора на строки операторского стола и фильтр “мои задачи” через общий слой `admin_task_assignments`.
- SLA/приоритеты задач в операторском столе: обычный/срочно/критично, дедлайны 24ч/48ч/7д/без срока, просрочка и сортировка по важности.
- Автоматические SLA-правила по типам задач: анкеты на модерации получают обычный приоритет и дедлайн 48ч, заявки заказчиков — срочный 24ч, объединение аккаунтов — срочный 48ч, safety-жалобы — критичный 24ч.
- SLA-сводка на back-office панели: просроченные задачи, критичные задачи и мои просроченные задачи.

Не хватает:
- Глубоких отдельных таблиц внутри back-office панели для пользователей, всех анкет, всех кастингов и подборок.
- Таблиц пользователей.
- Продвинутых массовых workflow: сохранение детального журнала попыток bulk-операций как отдельной сущности, повтор неудачных операций из истории, эскалации и уведомления по нарушению SLA.
- Расширения audit на будущие back-office зоны: пользователи, жалобы/споры, billing, ручные правки тарифов и роли.

## 11. Безопасность и приватность

Уже есть:
- Supabase Auth, RLS-политики, роли аккаунтов, разграничение admin/casting/user.
- Email/phone sign-in и подтверждение email/телефона в аккаунте.
- Удаление аккаунта и базовые safety/admin-зоны.

Не хватает:
- Публичных документов: Privacy Policy, Terms of Service, Cookie Policy и Consent/processing notice для моделей, родителей и заказчиков.
- Принятия документов при регистрации с сохранением версии политики, даты, IP/User-Agent и источника согласия.
- Проверки силы пароля при регистрации/смене: минимальная длина, запрет простых паролей, подсказки по надежности, блок популярных паролей.
- Rate limiting и анти-bruteforce для логина, восстановления пароля, OTP/SMS и повторной отправки email.
- 2FA/MFA для админов и casting/account-owner ролей, с резервными кодами.
- Управления активными сессиями: список устройств, “выйти со всех устройств”, уведомление о новом входе.
- Security notifications: новый вход, смена пароля/email/телефона, подозрительная активность.
- Приватности профиля и account tag: публичный/только участникам диалогов/скрытый режим, отдельные настройки видимости email/телефона.
- Audit trail для чувствительных действий пользователя: смена контактов, пароля, ролей, удаление аккаунта, экспорт данных.
- Data export/download my data и понятная процедура удаления персональных данных.
- Защиты медиа: signed URLs/ограничение публичности для приватных материалов, watermark/anti-scraping для чувствительных фото, где уместно.
- Content moderation/safety flow для жалоб, блокировок пользователей и скрытия подозрительных аккаунтов.
- Admin security hardening: MFA обязательно, ограничение опасных действий, reason/comment для ручных правок, журнал доступа.
